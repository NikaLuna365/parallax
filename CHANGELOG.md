## 0.37.4 — z.ai (GLM) verifier fallback provider
Adds **z.ai (GLM)** as a cross-model verifier provider so the independent verifier keeps working when the `codex` CLI is **unavailable or rate-limited** — the case that stalls a run today. z.ai speaks the **OpenAI-compatible HTTP API**, so it needs **no CLI** and works anywhere `curl` does (including laptop-off cloud routines). **Additive and opt-in — with no `.parallax/codex.toml` fallback configured, runtime is byte-for-byte v0.37.3.** No new public command, no benchmark / quality / production-readiness claim; every v0.31–v0.37.3 boundary preserved.

- **New provider `zai` (form `api`).** Provider values stay **directive** (consumed by `role-codex-judge`, not a machine enum): the chain is now `codex | gemini | zai`. The documented default chain is **codex → z.ai GLM**. An `api` provider carries a `base_url` and a `key_env` — the **NAME** of the env var holding the key, never the key.
- **Secret hygiene (unchanged rule, now with a place to put it).** The key is read from the env var `key_env` names (default `ZAI_API_KEY`) and **never** from the committed config. Put it in an **untracked** `.parallax/zai.env` and `source` it before the run; `.gitignore` now covers `*.env` / `.parallax/**/*.env` / `.parallax/zai.env`. The verifier **fails closed** if the env var is empty (never runs unauthenticated).
- **Invocation contract (`skills/role-codex-judge`).** The `api` path documents the exact z.ai call: `POST` the `base_url` with `Authorization: Bearer $ZAI_API_KEY` and `response_format={"type":"json_object"}`; the verdict JSON is `.choices[0].message.content`. GLM is a **reasoning model** — its chain-of-thought is in `.message.reasoning_content` (ignored) and `max_tokens` must be generous or `finish_reason:"length"` truncates the verdict. As with the Gemini path, the schema is **embedded in the prompt and the returned JSON validated locally against the FULL schema** (top-level-`allOf` schemas still go through `scripts/strip-openai-schema.py` for the call only).
- **Docs.** `assets/codex/codex.toml.example`, `README.md` (requirements + minimal config), `commands/run.md` and `commands/auto.md` now show the `codex → z.ai GLM` fallback. Endpoint + model id (`glm-5.2`) confirmed live against `api.z.ai` on 2026-07-07.
- **Harness.** New `[zai_provider]` section (directive presence checks: config example documents `zai`/`api`/`base_url`/`key_env`; role skill documents the Bearer call, `key_env`, and the reasoning-model `max_tokens` caveat) plus a **secret-leak guard** that scans the whole tree for a literal z.ai-key pattern and fails closed if one is ever committed. `claude plugin validate` passes.

As everywhere, **mechanical** = a script/schema the harness executes; **directive** = a prompt-contract obligation the harness can only check is present.

- **F1 Monorepo-aware blindfold guard (mechanical).** `scripts/blindfold-guard.py` gains a slice-scoped mode: `--scope-manifest` takes a per-slice manifest (`assets/blindfold-scope.schema.json` — new, `parallax-blindfold-scope-v1`) naming the slice's OWN new/changed `protected_impl_paths`/`protected_test_paths` (always fail closed on the opposite track, checked before every allowlist) plus package-specific `dependency_allow_globs` the test side may see for import resolution; in scoped mode the existing base tree is visible by design (that's what import resolution needs) while the slice's own impl — and its own package `dist/` — still fail closed. ## 0.37.1 — Finalize Freshness Patch
Closes the one accepted v0.37.0 follow-up (Codex rerun finding F1): `finalize-gate.py` documented a "fresh run-state" requirement but implemented it as only a non-empty `updated_at`. v0.37.1 makes freshness a real mechanical binding. **A patch — no new public command, no benchmark/quality claim**; every v0.31–v0.37.0 boundary preserved.

- **Terminal completion receipt (schema).** `assets/run-state.schema.json` gains a `completion` object — `completed_at`, `run_id`, `verified_tree`, `run_evidence_sha256`, `events_jsonl_sha256`, `terminal_event` (`run_completed`) — **required once `status == "complete"`**. It binds the run-state to the exact terminal evidence bundle without a self-referential dependency on the final commit oid.
- **Freshness-bound finalize gate.** `scripts/finalize-gate.py` now fails closed unless, read from the committed feature ref: the run-state is **schema-valid**; `updated_at` *and* `completion.completed_at` parse as **ISO-8601** (a bare `"t"` is rejected); `completion.run_id`/`verified_tree` match the run-state and `verified_tree` equals the recomputed `code-tree-hash`; the committed `run-evidence.json` validates and its `run.run_id`/`slug`/`status` match; every `events.jsonl` line validates and a same-run **`run_completed`** event exists; the **sha256** of the committed `run-evidence.json` and `events.jsonl` equal `completion.*_sha256`; and a non-null `lock` belongs to the same `run_id`. The v0.37.0 requirements (no `green-unverified`, per-slice green arbiter receipts, `epic-gate.py` delegation) still hold; missing `jsonschema` remains fail-closed.
- **Finalization order.** `commands/run.md` Step 4 spells out the terminal order (drain debt → `run-evidence` complete → append `run_completed` → sha256 evidence → write the `completion` receipt → commit the bundle → pin OID → finalize-gate → push) and states that a present but unbound `updated_at` is not freshness.
- **Harness.** `tests/t_finalize_gate.sh` builds a self-consistent completion bundle and adds adversarial cases: no completion (the old weak path), non-ISO timestamp, evidence `run_id` mismatch, missing `run_completed` event, evidence byte-tamper, and `verified_tree`≠`code-tree-hash` — all hold; `tests/run.sh` gains a `[finalize_freshness]` section. ## 0.31.0 — safe completion (`/parallax:resolve`) + an Existing Affordance Review
v0.30 reliably **stopped** on a real spec-gap (a `safe-resolution`) but couldn't turn the human's decision into a verified, finished feature; an earlier failure mode also let a correct build freeze a **new subsystem the codebase already had a seam for** ("green overbuild"). v0.31 closes both — the first with a new contract-generation cycle, the second with a prompt-contract review in `/parallax:spec`.

- **Safe completion — a human decision becomes a new contract generation, never a "ship anyway".** New command **`/parallax:resolve <slug>`** (`commands/resolve.md`): for a run parked at `status=needs-resolution`, it shows the open *structured* decisions, takes one decision per item (`choose-option` / `custom-rule` / `rescope` / `abandon` — never `ignore`/`ship-anyway`/manual-`fixed`), builds a contract-only candidate, requires an **exact one-time token**, and applies the batch atomically. Then it **fully invalidates** the old certification and rebuilds: a fresh epic base, all slices `pending`, the old code/tests/ledgers gone from the active tree (kept only in `history/generation-N/`), fresh pre-freeze review, fresh blind tracks.
  - **Mechanical (scripts + schemas + harness-executed):** `scripts/resolution.py` is the **only** writer of the resolution queue, batch receipts, and `feature-state.json` generation transitions — fail-closed on a missing validator, stale source hash, reused/mismatched one-time token, empty contract diff, unclosed blocking item, unsupported parked reason, or a non-`+1` generation. `scripts/generation-restart.sh` performs the **append-only** restart: a restart commit parented on `[old-feature-tip, fresh-epic]` (so the branch only ever fast-forwards — never a force-push), the feature ref advanced by an atomic `git update-ref` CAS (exactly one racing resolver wins; the loser refuses to clobber), idempotent across a crash before/after the CAS. `scripts/epic-gate.py` requires the promoted run to be the **active** generation (`feature-state.status=complete`, run-state generation == active generation, `run_id` == `active_run_id`, continuous resolution chain) — so a stale-generation green ledger can never certify the new contract. New schemas: `feature-state`, `resolution-queue`, `resolution-receipt`; `run-state` gains `feature_id` / `contract_generation` / `parent_run_id` / `resolution_queue` / `resolution_chain` and the `needs-resolution` / `resolving` / `abandoned` statuses.
  - **Directives (orchestrator-executed):** `resolve.md` presenting the token to the human and reading the reply, building the candidate contract, and running the fresh self-review + pre-freeze; `run.md` / `auto.md` recording a **structured** resolution-queue item at a spec-gap park (via `resolution.py add-item`) and flipping the run to `needs-resolution`; `role-arbiter` emitting a spec-gap as competing readings + spec refs; `role-codex-judge` treating a `spec-gap`/`under-specified` finding as a resolution-item source keyed by its stable ledger id. `escalations.md` stays a human-readable projection — the structured queue is the authoritative source.
  - **Backward compatibility (P3).** `resolution.py migrate` synthesizes a generation-1 `feature-state` from a v0.30 run-state, assigns a `feature_id`, and stamps `contract_generation=1` — **idempotent** (a second run is a validated no-op) and **fail-closed** (a missing/garbled run-state escalates; a free-text `escalations.md` is never an authoritative source). A missing `contract_generation` is treated as generation 1.
  - **Residual (unchanged, stated plainly).** The one-time token is an **auditable consent marker, not a cryptographic proof of who typed it**; a perfectly self-consistent forged commit is still covered only by the Step-5 commit inventory and the human `epic → main` PR + CI. v0.31 invalidation is intentionally **all-slices** (no partial-impact optimization) — deliberately expensive, because reusing old certification against a new contract is exactly the hole this release closes.

- **Existing Affordance Review (prompt/contract patch — no new command/script/schema).** `/parallax:spec` now hunts existing seams during Step 1, runs a mandatory **Step 3.5 Existing Affordance Review** before proposing approaches (per candidate: covers the behavior fully? exact small change? what stays uncovered? what risk?), orders approaches **thin-overlay → small-local-change → new-module** (a non-viable overlay is still shown `rejected`, so a skipped check is visible), records the result in a frozen-spec **`Existing affordance review`** section, and adds an **always-applies** self-review pass (a new subsystem with no recorded rejection of plausible affordances is a spec blocker). The pre-freeze reviewer (`spec.md` Step 9 + `role-codex-judge`) now flags **unjustified overbuild** classified by repo evidence (`high`/`medium`/`low`), never by style. The short path never relaxes a safety/ambiguity/validation gate and is **not** a line-count rule; in autonomous mode it's a mechanical choice only (a product-behaviour change parks/escalates). New `[review_failclosed]` check runs triage with the validator hidden and asserts `escalate`.
- **P1#4 — fingerprints no longer collapse distinct defects.** The merge keyed solely on `sha256(kind|spec_ref|file)`, so two *different* violations of the same section in the same file merged into one ledger entry and the first was **lost** (reproduced: 2 findings in → 1 stored). `merge-ledger.py` now matches two-pass: a round finding/resolution may cite an **exact ledger `id`** (binding precisely even when fingerprints collide), and uncited findings take the first *still-unconsumed* same-fingerprint slot else become a new distinct id — so two same-section defects never collapse. `review-round.schema.json` gains an optional `id` on `findings` and `resolved`; `role-codex-judge` is told to echo it. The cross-round rephrase→same-id regression behavior is unchanged (still tested). New `[merge_ledger_collision]` check.
- **P1#5 — an UNVERIFIED run cannot advance the append-only epic.** Under `on_missing = "warn"` the run proceeded stamping outputs `UNVERIFIED`, but Step 4 still auto-advanced feature → epic. Fixed in `run.md`: a hard verification gate (`PARALLAX_VERIFIED`) precedes the epic push — an UNVERIFIED run may push the **feature** branch for human review but **parks an epic-hold** and never advances the append-only epic; only a fully verified feature advances automatically. (`epic → main` remains a human PR + CI, as before.) New `[epic_verify_gate]` presence check; this guard is a **directive** executed by the orchestrator (honestly labeled), with a concrete bash gate that `bash -n` validates.
- Harness → **40 executable/presence checks** (32 → 40; +8 for the five findings above). All prior checks still pass.
- **Honest scope:** P0#1/#2/#3 and P1#4 are **mechanical** (executed by scripts/git and locked by the harness). P1#5 is a **directive** the orchestrating model must honor — the harness checks the gate is present and bash-valid, not that the model runs it; the same honesty split as `on_missing` and `mode` semantics.

## 0.21.0 — fourth-audit remediation: close the producer-self-certification holes
The v0.20 review ledger was written by the orchestrating Claude and triaged with a ledger-supplied policy — so a crafted ledger could mechanically turn a `high safety` finding into `green`. v0.21 makes certification something only the *independent* model can grant. (All findings reproduced before fixing.)
- **P0 — policy comes only from trusted config.** `triage.py` now reads the `[review]` policy **only** from the `.parallax/codex.toml` passed via `--policy`, **never** from the ledger (a ledger `policy` with `always_block_kinds=[]` previously waved a `safety` finding through). Missing/unparseable config ⇒ strict built-in defaults (fail-closed). The ledger schema also **rejects** any `policy` key (second layer). *(reproduced: high-safety → green; now → block.)*
- **P0 — a fix must be proven, not asserted.** A `fixed` finding counts as settled **only** if `verified_by == "codex"` **and** `last_verified_diff == --current-diff` (the actual assembled tree). A `fixed` Claude merely stamped, or one verified against a stale tree, is treated as **live** and still blocks. Enforced in `triage.py` **and** in the ledger schema (status=fixed ⇒ requires `verified_by:"codex"` + a hex `last_verified_diff`). *(reproduced: faked `fixed` → green; now → block.)*
- **P0 — unified review-round + mechanical merge (the producer authors nothing).** New `assets/codex/review-round.schema.json` is what the verifier emits (`findings` + positively-re-verified `resolved`). New `scripts/merge-ledger.py` is the **only** writer of the ledger: it maps the round in by **fingerprint** (`sha256(kind|spec_ref|file)` → the same defect keeps one id across rounds), assigns ids, and sets `verified_by=codex` only on the verifier's `resolved` items. Claude never invents `spec_ref`/`evidence`/`id`/lifecycle. Resolves the old verdict↔ledger shape mismatch and the missing finding lifecycle.
- **P1 — per-slice ledgers, single-sourced budget.** Ledgers moved to `.parallax/<slug>/reviews/<slice_id>.json` (one per slice — a slice can't spend another's budget). `rounds_used` lives **only** in the ledger (run-state points at the file via `reviews_dir`; the divergent `review_rounds` counter is gone). Round semantics pinned: one `merge-ledger` call = one round, the initial review is round 1, so `max_rounds=2` ⇒ at most two verifier invocations.
- **P1 — parallel correctness.** The arbiter **and** the post-green verifier now run with cwd = the slice's `$WT/S<n>/assembly` (the tree actually under review), never `$ROOT`. And `feature/<slug>` is **never checked out** in a persistent worktree during a wave (`$ROOT` detached) — moving the branch ref via CAS while it was checked out left that worktree stale/dirty (`D src/…`); reproduced and fixed.
- **P1 — expired-lock steal is now race-safe.** Stealing an expired lease uses `--force-with-lease=<ref>:<observed-oid>` (a bare `--force` let *two* stealers both win — reproduced), and release is **fenced** to your own lock oid so you never clobber a successor. `t_lock.sh` executes both races.
- **P2 — verdict/round consistency.** `verdict.schema.json` and `review-round.schema.json` now reject `pass`+non-empty-findings and `concerns`+empty-findings.
- Harness → **32 executable checks** (added: the four reproduced exploits as regressions — ledger-policy ignored, faked-fixed blocks, codex-verified-vs-current-diff settles, stale-verify re-blocks; `merge-ledger` fingerprint/lifecycle; schema rejects fixed-without-proof + policy-bearing ledger + verdict inconsistency; expired-lock lease steal + fenced release).

## 0.20.0 — verifier memory: anti-oscillation review ledger + bounded hybrid disposition
Addresses the cross-model verifier's missing memory between review rounds (it re-discovered fixed findings, never confirmed a fix held, and any `low` nit could stall the whole dependent DAG).
- **Review memory is a ledger, not a session.** Each review is a **fresh** verifier (`[review] resume_codex_session = false`); memory lives in a branch-committed `.parallax/<slug>/review-ledger.json` (new `assets/codex/review-ledger.schema.json`). A fresh verifier re-checks prior `open`+`fixed` findings for **regression first**, then scans for new ones — killing the `n → fix → m → variant-of-n` loop **without** the anchoring a persistent `codex exec resume` session causes (and the non-interactive session id is fragile to capture and wouldn't survive a cloud fresh-clone). Every finding must carry a **`spec_ref` + concrete evidence**; style/preference is not a defect. (Verified: `codex exec resume`/`--last` exist — we deliberately don't use them.)
- **Disposition is mechanical, not editorial — `scripts/triage.py`.** A new, unit-tested pure function decides `green | block | escalate` from the ledger + `[review]` policy: `low` = **advisory** (recorded, non-blocking); `medium`/`high` **block**; `safety`/`anti-cheat`/`spec-gap` and any **reproducible functional error** **always** block. Claude may **contest** a blocker only via a formal, ledgered reason (`duplicate`/`not-reproducible`/`contradicts-spec`/`out-of-scope`) — a contested medium/high **escalates** (human decides), it is **never silently dropped or auto-greened**. So the asymmetry that mattered (Claude can't quietly overrule the verifier on anything that blocks) is preserved; the only relaxation is the `low`-advisory release valve.
- **Bounded loop.** `[review].max_rounds` (default 2) caps review rounds per slice (tracked as `review_rounds` in the checkpoint, distinct from the 3-iteration worker breaker); at the cap with blockers still live, the slice **parks** instead of looping on ever-smaller nits.
- Contracts updated: `role-codex-judge` (the ledger protocol + hybrid disposition, honestly split into mechanical vs directive), `run.md` Step 2c (*Review memory, rounds & disposition* — ledger merge → triage → block/escalate/green), `parallax-core` (the no-quiet-overrule nuance), run-state (`review_rounds` per slice + `review_ledger` queue path), README.
- Harness → **30 executable checks** (added: `triage.py` disposition matrix incl. exit codes, review-ledger schema rejects missing `spec_ref`/bad kind/bad rebuttal, `[review]` config semantics, contract-presence).

## 0.19.0 — third-audit remediation (cloud-lock race, transactional parallel integration, exact-resume schema)
- **#1 (P0 cloud lock, fixed):** two fresh same-`HEAD` clones used to *both* win the lock — pointing the ref at `HEAD` made the loser's "create" a no-op same-value push. The lock now points at a **unique lock commit** (`git commit-tree` with the `run_id` baked in) and is taken create-if-absent locally / by **`git push --force-with-lease=<ref>:`** in the cloud (push only if the ref is absent on origin). `t_lock.sh` now models the *real* same-HEAD race and includes a guard proving the old same-value approach let both win.
- **#2/#3 (P1 parallel integration, fixed):** each slice integrates in its **own `$WT/S<n>/assembly` worktree** (not the shared `feature/<slug>` tree), **transactionally** — both track patches apply all-or-nothing; a conflict triggers `git reset --hard` so the assembly is abandoned and `feature/<slug>` is **never** left half-patched. Advancing `feature/<slug>` is a **CAS `update-ref`** (old-value = the tip we forked from), serializing concurrent integrations; a slice that loses the race re-applies its `wave_base` diff onto the new tip. New `t_conflict.sh` executes the conflict → rollback → feature-unmoved path.
- **#4 (P1 binary files, fixed):** integration uses **`git diff --binary … | git apply --3way --index --binary`** — a plain text diff of a binary file fails (`cannot apply binary patch without full index line`). New `t_binary.sh` proves the `--binary` path applies (bytes match) and the plain path fails.
- **#5 (P1 exact resume, fixed):** run-state schema now records **`wave_base`** per slice (required once `in_progress`/`green-unverified`), constrains all object ids (`base_tip`/`code_tip`/`test_tip`/`wave_base`/`verified_diff`) to a **hex-SHA pattern**, and conditionally requires **`lock` when `status=running`** and **`paused` when `status=paused-on-limit`**. The harness validates a complete checkpoint and rejects 5 distinct incomplete/invalid ones.
- **#6 (P1 sole verdict, fixed):** the verdict schema gains **`code-fault`/`test-fault`** finding kinds, and `role-codex-judge` documents **sole-mode RED arbitration** (the verifier classifies the fault, a third behavior beyond pre-freeze/post-green) so `mode = sole` is actually expressible in the schema it emits.
- **#7 (P2 branch prefix, fixed):** `${PREFIX}` now drives the **sibling value-scan** (Step 0.6) and the **commit inventory** (Step 5) too — a `claude/` cloud run scans/report `claude/*`, not a hardcoded `feature/*`.
- **#8 (P2 timeouts, made real + honest):** verifier calls are wrapped in a real wall-clock **`timeout "$TIMEOUT_S"`** (`curl --max-time` for the api form); exit 124 → next provider. The contract now states plainly which guards are **mechanical** (timeout, next-provider-on-nonzero) vs **directives the agent executes** (the in-process retry budget, the api-form curl) — the harness can only check the mechanical ones.
- Harness → **25 executable checks** (added binary integration, transactional conflict-rollback, assembly-worktree doc, force-with-lease same-HEAD lock, 5-way schema rejection, verdict kinds, real-timeout wrappers).
