# Changelog

All notable changes to the Parallax plugin. Versions are cumulative.

## 0.33.0 — optional Project Scout fanout in `/parallax:spec` (large repos)
On a large or unfamiliar repo, Phase 1 must find existing seams, local architecture contracts, real testing seams, and source-of-truth domain logic before freeze — and doing it all linearly in one agent floods context and misses things. v0.33 lets the main `/parallax:spec` agent optionally delegate the **read-only evidence search** to a few **bounded internal scouts**, then verify and decide itself. Same prompt-contract pattern as v0.31/v0.32: **no new public command, no new schema/state, no change to the epic/main gates or to v0.31 safe-completion / v0.32 Architecture Fitness**.

- **Scouts gather evidence; they never decide.** A new pipeline-internal agent + role (`agents/project-scout.md`, `skills/role-project-scout/SKILL.md`) is **read-only** (no `Write`/`Edit` tools): it searches **one bounded lens** — existing affordances, local architecture contracts, testing/validation seams, source-of-truth/domain logic, or risky integration points — and returns a **compact report** with `file:line` evidence, confidence, uncertainty, and a *recommended main-agent verification* list. It cannot edit files, ask the user, freeze/resolve, run `/parallax:run`, or make an architecture/scope decision, and it reports "not found" honestly instead of inventing a seam.
- **The main agent owns Phase 1 and verifies before relying.** `commands/spec.md` gains an **optional Step 1.5 Project Scout Fanout** (used only when the runtime supports sub-agents *and* the repo is large/unknown — `git ls-files` > ~300, monorepo shape, cross-cutting or unknown-surface work; otherwise linear) and a **Step 1.6 verification rule**: before freeze the main agent must directly open and confirm every scout finding it relies on (the chosen shape, used/rejected affordances, cited architecture contracts, validation commands/globs, domain source-of-truth). "The scout found X" is not evidence unless the main agent checked X; the Existing Affordance Review and Architecture Fitness stay the main agent's decisions. The frozen spec gains an optional **`## Project scout evidence`** section recording whether fanout ran and which evidence was verified.
- **Bounded, linear-default, fresh per generation.** Fanout is capped (2–5 scouts, bounded scope/output) — a context-saving tool, not a repo-wide audit; the **linear v0.32 flow is the default** and small repos are never blocked or slowed. `/parallax:auto` may fan out only for *mechanical* evidence and still parks a principled product fork. After `/parallax:resolve`, scout evidence is **stale by default** — old reports are history only and can never certify the new generation, which re-gathers fresh.
- **Mechanical vs directive (honest split).** As in v0.31/v0.32, v0.33 adds **no new mechanical enforcement** — the fanout, the read-only boundary, and the verify-before-rely rule are all **prompt-contract directives**. The harness mechanically locks that the directives are present and that the forbidden things did not appear: `tests/run.sh` gains `[project_scout]` (the spec step/verification/section, the read-only no-decision scout role, the agent loading it with no write tools, **no** public scout command, and v0.31/v0.32 material preserved). Behaviour is LLM-judged via `tests/project-scout-eval-cases.md` (9 cases incl. small-repo + runtime-unavailable fallbacks, a hallucination rejection, a scout-overreach rejection, and resolve-generation freshness).
- **Harness → 85 checks (79 → 85), 0 failed; `claude plugin validate` passes.** Built on `parallax_v032_work` (v0.32.0); `parallax_push` untouched. Honest scope: v0.33 does **not** claim fanout always improves quality — the hypothesis (bounded fanout gathers better evidence on large repos at lower context cost while the main agent keeps the decision) is for a documented targeted eval, not a claimed result; the market benchmark waits for v0.36.

## 0.32.0 — Architecture Fitness in `/parallax:spec`
A narrow, pre-freeze maintainability gate added to `/parallax:spec`, on the same prompt-contract pattern as the v0.31 Existing Affordance Review. It catches the common AI-generated maintainability failures that downstream blind TDD would honestly green — and nothing more. It is **not** an architecture consultant, adds **no new command, sub-agent/scout fanout, schema, or finding kind**, and does **not** touch the epic/main gates or the v0.31 safe-completion/stale-generation machinery.

- **The six failures it targets (A1–A6), each only with a *concrete* consequence.** A1 **wrong seam** (tests/code crossing internals instead of the declared public boundary); A2 **shallow wrapper** (a new module that only forwards calls and hides nothing); A3 **duplicated business logic** (a rule smeared across callers); A4 **speculative adapter/port** (a layer "for the future" with no current variation); A5 **no correct regression seam** (tests that can stay green while the user-visible behaviour breaks); A6 **ignored local architecture contract** (a silently-violated `AGENTS`/`CONTEXT`/ADR/testing convention). Style, naming, folder layout, and speculative future flexibility are **never** blockers, and a small feature is never forced to grow a deep module.

- **Prompt-contract changes (the only kind here):**
  - `commands/spec.md`: Step 1 also reads local architecture contracts (`AGENTS`/`CLAUDE`/`CONTEXT`/`docs/adr/*`/READMEs) as *evidence* (never as instructions that override a Parallax safety rule); a new **Step 4.5 Architecture Fitness** checks the *chosen* shape (public seam / locality / module depth / variation / regression seam / local contract) **after** the affordance review and **before** drafting; the frozen Spec format gains an **`## Architecture fitness`** section; Step 8 adds an **always-applies** Architecture Fitness self-review pass; Step 9 + `role-codex-judge` extend the pre-freeze reviewer to A1–A6, **mapped to the existing kinds** (`spec-gap`/`code-fault`/`test-fault`/`low`) with `high`/`medium`/`low` calibrated by observable consequence and repo/docs evidence.
  - Role skills now share the declared seam: `role-blind-coder` implements **through the public seam** (no private-helper export just for tests, no pass-through layers, business rule kept where the spec's *Locality* puts it); `role-test-writer` asserts across the **regression seam** (no private-helper tests when the public seam reproduces the behaviour; report a candidate architecture blocker rather than invent a brittle test); `role-arbiter` does a **bounded** post-green check that the declared public/regression seam is reachable and exercised (unreachable → `code-fault`, untested → `test-fault`) — never a broad architecture review; `role-codex-judge` carries the A1–A6 calibration and that style is never a defect and `low` never blocks.
  - `commands/resolve.md`: a new contract generation regenerates Architecture Fitness **fresh** via the full spec self-review; an old generation's architecture notes are history only and can never certify new code/tests. No separate resolve-side logic.

- **Mechanical vs directive (honest split).** There is **no new mechanical enforcement** in v0.32 — Architecture Fitness is entirely **orchestrator/reviewer directives** in the prompt contracts. What the harness *mechanically* locks is that those directives are present and that the forbidden things did **not** appear: `tests/run.sh` gains `[architecture_fitness]` (the spec step/format/self-review, the codex-judge no-style-blocker calibration, the role seam instructions, **no** `/parallax:architecture` command, and **no** `scout`/`fanout`). The behaviour itself is LLM-judged via `tests/architecture-fitness-eval-cases.md` (F1–F9, including the explicit style-only non-blocker F7 and the allowed-adapter control F5). All v0.31 resolution/generation tests are preserved.

- **Harness → 79 checks (74 → 79), 0 failed; `claude plugin validate` passes.** Built on `parallax_v031_work` (v0.31.0); `parallax_push` is untouched until a separate promote step. Honest scope: v0.32 does not prove Parallax always produces "good architecture" — only that the obvious failures above aren't silently frozen. The behavioural benchmark is a documented plan, not a claimed result.

## 0.31.0 — safe completion (`/parallax:resolve`) + an Existing Affordance Review
v0.30 reliably **stopped** on a real spec-gap (a `safe-resolution`) but couldn't turn the human's decision into a verified, finished feature; an earlier failure mode also let a correct build freeze a **new subsystem the codebase already had a seam for** ("green overbuild"). v0.31 closes both — the first with a new contract-generation cycle, the second with a prompt-contract review in `/parallax:spec`.

- **Safe completion — a human decision becomes a new contract generation, never a "ship anyway".** New command **`/parallax:resolve <slug>`** (`commands/resolve.md`): for a run parked at `status=needs-resolution`, it shows the open *structured* decisions, takes one decision per item (`choose-option` / `custom-rule` / `rescope` / `abandon` — never `ignore`/`ship-anyway`/manual-`fixed`), builds a contract-only candidate, requires an **exact one-time token**, and applies the batch atomically. Then it **fully invalidates** the old certification and rebuilds: a fresh epic base, all slices `pending`, the old code/tests/ledgers gone from the active tree (kept only in `history/generation-N/`), fresh pre-freeze review, fresh blind tracks.
  - **Mechanical (scripts + schemas + harness-executed):** `scripts/resolution.py` is the **only** writer of the resolution queue, batch receipts, and `feature-state.json` generation transitions — fail-closed on a missing validator, stale source hash, reused/mismatched one-time token, empty contract diff, unclosed blocking item, unsupported parked reason, or a non-`+1` generation. `scripts/generation-restart.sh` performs the **append-only** restart: a restart commit parented on `[old-feature-tip, fresh-epic]` (so the branch only ever fast-forwards — never a force-push), the feature ref advanced by an atomic `git update-ref` CAS (exactly one racing resolver wins; the loser refuses to clobber), idempotent across a crash before/after the CAS. `scripts/epic-gate.py` requires the promoted run to be the **active** generation (`feature-state.status=complete`, run-state generation == active generation, `run_id` == `active_run_id`, continuous resolution chain) — so a stale-generation green ledger can never certify the new contract. New schemas: `feature-state`, `resolution-queue`, `resolution-receipt`; `run-state` gains `feature_id` / `contract_generation` / `parent_run_id` / `resolution_queue` / `resolution_chain` and the `needs-resolution` / `resolving` / `abandoned` statuses.
  - **Directives (orchestrator-executed):** `resolve.md` presenting the token to the human and reading the reply, building the candidate contract, and running the fresh self-review + pre-freeze; `run.md` / `auto.md` recording a **structured** resolution-queue item at a spec-gap park (via `resolution.py add-item`) and flipping the run to `needs-resolution`; `role-arbiter` emitting a spec-gap as competing readings + spec refs; `role-codex-judge` treating a `spec-gap`/`under-specified` finding as a resolution-item source keyed by its stable ledger id. `escalations.md` stays a human-readable projection — the structured queue is the authoritative source.
  - **Backward compatibility (P3).** `resolution.py migrate` synthesizes a generation-1 `feature-state` from a v0.30 run-state, assigns a `feature_id`, and stamps `contract_generation=1` — **idempotent** (a second run is a validated no-op) and **fail-closed** (a missing/garbled run-state escalates; a free-text `escalations.md` is never an authoritative source). A missing `contract_generation` is treated as generation 1.
  - **Residual (unchanged, stated plainly).** The one-time token is an **auditable consent marker, not a cryptographic proof of who typed it**; a perfectly self-consistent forged commit is still covered only by the Step-5 commit inventory and the human `epic → main` PR + CI. v0.31 invalidation is intentionally **all-slices** (no partial-impact optimization) — deliberately expensive, because reusing old certification against a new contract is exactly the hole this release closes.

- **Existing Affordance Review (prompt/contract patch — no new command/script/schema).** `/parallax:spec` now hunts existing seams during Step 1, runs a mandatory **Step 3.5 Existing Affordance Review** before proposing approaches (per candidate: covers the behavior fully? exact small change? what stays uncovered? what risk?), orders approaches **thin-overlay → small-local-change → new-module** (a non-viable overlay is still shown `rejected`, so a skipped check is visible), records the result in a frozen-spec **`Existing affordance review`** section, and adds an **always-applies** self-review pass (a new subsystem with no recorded rejection of plausible affordances is a spec blocker). The pre-freeze reviewer (`spec.md` Step 9 + `role-codex-judge`) now flags **unjustified overbuild** classified by repo evidence (`high`/`medium`/`low`), never by style. The short path never relaxes a safety/ambiguity/validation gate and is **not** a line-count rule; in autonomous mode it's a mechanical choice only (a product-behaviour change parks/escalates). Behavioural eval set `tests/affordance-eval-cases.md` (E1–E6). Safe-completion compatibility holds by construction: `/parallax:resolve` re-runs the full spec self-review per generation, so the affordance section regenerates and the old one stays in `history/`.

- **Harness → 73 executable/presence checks (55 → 73), 0 failed; `claude plugin validate` passes.** New executable git/unit tests: `t_resolution_gate` (P1: one-time token, +1 generation, fail-closed set, stale-generation hold), `t_resolution_generation` + `t_resolution_race` (P2: append-only restart, no old code on active paths, history archive, fast-forward publish, idempotency, atomic CAS race), `t_resolution_migration` (P3: idempotent, resolve-ready, fail-closed); new presence sections `[resolution_command]` and `[affordance_review]`; `[shell_syntax]` now `bash -n`s `resolve.md` too. ZIP = `git archive` of the tag (tracked files only, no `.pyc`).

- **Honest verification gap.** The AC9 **safe-completion benchmark** (≥15 live resolution cycles proving `safe_completion=1.0` / `false_green=0`, clear-task regression unchanged) requires a real model pipeline and is **not** run by the unit harness — it is documented as a runnable plan in `tests/safe-completion-benchmark.md` and remains the outstanding empirical validation, recorded as a gap rather than claimed as a pass.

## 0.30.0 — bounded pre-freeze review: stop reviewer churn without false-green skipping
An interactive money-critical run reproduced a hole outside the existing post-green ledger: pre-freeze Codex rounds went `41 -> 16 -> 11 -> 12`. `[review].max_rounds=2` was reached and reported, but it governed only per-slice post-green triage. The orchestrator then treated the owner's answer to an unrelated product fork as an implicit round-3 budget extension and launched round 4 without another explicit grant. The stored `pre_freeze.round*.json` files were Claude-authored summaries rather than the verifier's raw schema-valid receipts. The late findings were not all nits (null degradation, negative included volume, and fallback/re-export bypasses were material), so blindly ignoring them would replace an endless loop with a false green.
- **P0 — executable pre-freeze budget.** New `scripts/pre-freeze-budget.py` + `assets/codex/pre-freeze-state.schema.json` make `.parallax/<slug>/reviews/pre-freeze-state.json` the single source of truth. `check` must authorize every dispatch; `record` validates the provider's raw `spec-adversary` JSON, writes the canonical immutable round receipt **and a snapshot/hash of the exact spec/slices/validation contract it reviewed**, records severity counts, and increments the round exactly once. Missing/tampered receipts or snapshots fail closed. `[review].pre_freeze_max_rounds` defaults to 2 (older configs fall back to `max_rounds`). At the cap the gate exits 2/checkpoints instead of polishing indefinitely.
- **P0 — extensions are explicit and one-shot.** The checkpoint emits `PARALLAX-GRANT:<slug>:pre-freeze-round-N`. Only `grant-one` with that exact human-repeated token authorizes round N; an unrelated product answer, vague approval, or silence cannot count. Each token grants one round only, so a `concerns` round checkpoints again before another invocation. The policy file is hash-pinned on first use; increasing the TOML limit mid-run escalates rather than resetting the budget. Autonomous mode can never self-grant and parks at the cap.
- **Audit receipts, not producer summaries.** The gate persists the verifier's raw schema-valid verdicts as `pre_freeze.round<N>.json`; Claude-authored trend summaries are no longer certification artifacts. State semantics reject non-sequential rounds/grants, mismatched counts, and unauthorized inventory.
- **Reviewer calibration.** The pre-freeze schema and `role-codex-judge` now require an observable consequence for a blocker: divergent external behavior, a false-green validation path, or concrete safety/money/data-integrity harm. Preferred helper names, source lines, internal control flow, or exhaustive matrices are `low`/inadmissible when behavior is already pinned. Real late defects remain blocking; diminishing returns trigger a checkpoint, never an automatic skip.
- **Regression harness.** New executable scenario proves: two base rounds; round 3 denied; an unrelated owner answer denied; exact token grants only round 3; round 4 denied again; policy drift denied; raw receipts/state remain sequential. Full harness: **55 passed, 0 failed** with `jsonschema`.
- **Honest scope.** Budget/state/schema enforcement is mechanical once `pre-freeze-budget.py` is invoked. Calling `check` before each provider and presenting the token to the human remain orchestrator directives; the token is an auditable explicit-consent marker, not cryptographic proof that a human typed it. The contract, autonomous command, judge role, README, and executable harness all wire and lock that boundary.

## 0.28.0 — eleventh-audit remediation: the stamped contract_hash must equal what the verifier read (worktree == HEAD)
v0.27 bound `contract_hash` (computed from HEAD) into the receipt and froze it per run — but the verifier reads the **worktree** copy of the contract, and the cleanliness guard covered only `src`/`tests`. So an **uncommitted** edit to `spec.md`/`validation.md` in the assembly worktree could be what Codex actually reviewed, while `contract-hash.sh HEAD` stamped the *committed* (strict) spec. Reproduced: worktree spec = weak B, committed spec = strict A, ledger stamped hash(A), gate `verified` — Codex reviewed against B. Same class as the v0.21/v0.22 P0#1 "the hash must represent what the verifier actually read," now applied to the contract files.
- **P0 — worktree-contract guard.** `run.md` 2c now asserts, before computing/stamping `contract_hash` (and the same check is required before dispatching the verifier), that the frozen contract in the assembly worktree is **identical to HEAD**: `git -C "$ASSEMBLED" diff --quiet HEAD -- .parallax/$SLUG/{spec.md,slices.md,validation.md,slices.lock}` (catches staged **and** unstaged drift) **plus** `ls-files --others` empty for those paths (catches an untracked contract file). Any drift ⇒ escalate, never stamp. Since `contract-hash.sh` hashes HEAD and the worktree now provably equals HEAD for the contract, the stamped `contract_hash` represents exactly the spec the verifier read. (The existing `src`/`tests` guard is vs the **index**, because the assembly intentionally stages the slice's code; the contract is frozen, so its guard is vs **HEAD**.)
- New `tests/t_contract_guard.sh` executes the git mechanic: an uncommitted spec edit is missed by the `src`/`tests` guard but caught by `git diff --quiet HEAD -- <contract>`, and an untracked contract file is caught by `ls-files --others`. Harness → **54 executable/presence checks** (52 → 54). All prior checks pass; ZIP = `git archive` of the tag (no `.pyc`).
- **Honest scope.** The guard is a git **directive** in `run.md` executed by the orchestrator; the harness executes the exact git mechanic (`t_contract_guard.sh`) and checks the contract wires it. Combined with v0.27's committed-`contract_hash` gate, the spec the verifier reads, the spec recorded in the receipt, and the spec in the promoted commit are now the same object. The irreducible residual (a fully self-consistent forged commit) is unchanged — covered by the Step 5 commit-inventory and the human `epic → main` PR + CI.

## 0.27.0 — tenth-audit remediation: the frozen spec contract is bound to the receipt
v0.26 pinned the code, policy, slice-set and slug to the promoted commit — but `code-tree-hash.sh` excludes **all** of `.parallax/`, so the frozen normative spec (`spec.md`, `slices.md`, `validation.md`, `slices.lock`) was never bound to the receipt. The colleague's tenth pass reproduced the gap: implement against a strict spec, get `verified`, then **rewrite `spec.md` and weaken `validation.md` after review** — `verified_tree` is unchanged (those files live under `.parallax/`) and the gate still returned `verified`. So an implementation could be checked against one spec and promoted carrying a different one. (Reproduced before fixing.)
- **P0 — a frozen `contract_hash` binds the normative spec to every receipt.** New `scripts/contract-hash.sh` computes a canonical hash of `.parallax/<slug>/{spec.md,slices.md,validation.md,slices.lock}` at a git ref (a missing file hashes a stable `<<MISSING>>` marker, so deletion is caught too). `scripts/merge-ledger.py --contract-hash` records it into the ledger and **freezes** it per run exactly like `policy_hash` — a mid-run spec/validation change ⇒ exit non-zero ⇒ the orchestrator PARKs and requires a fresh full review. `scripts/epic-gate.py` recomputes the contract hash from the **pinned committed commit** and requires every ledger's `contract_hash` to equal it; a mismatch ⇒ **hold**. `run.md` 2c computes and passes `--contract-hash`; `review-ledger.schema.json` gains the `contract_hash` field.
- This completes the pinning pattern: every input the verdict depends on — code+tests (`verified_tree`), the `[review]` policy (`policy_hash`), the slice-set (`slices.lock`), the slug, **and now the spec/validation contract** (`contract_hash`) — is bound to the one immutable commit the gate checks and pushes (`VERIFIED_OID`). `code-tree-hash.sh` still excludes `.parallax/` (correct — it ignores mutable ledger/run-state churn); the normative `.parallax/` artifacts are covered by `contract_hash` + `policy_hash` instead.
- Harness → **52 executable/presence checks** (50 → 52: +`[contract_freeze]`, +the `contract-mutated-after-review` scenario in `t_epic_gate.sh`). All prior checks pass; ZIP = `git archive` of the tag (no `.pyc`).
- **Honest scope & residual.** `contract_hash` (freeze + gate) is mechanical (`merge-ledger.py`/`epic-gate.py`, harness-executed). Computing and passing it is an orchestrator **directive** in `run.md`. The inherent residual is unchanged and now genuinely minimal: a *perfectly self-consistent forged* feature commit — code, tests, spec, validation, slices.lock, run-state, ledgers and the committed policy all mutually agreeing — cannot be distinguished from a genuine one without re-running the verifier; that is covered by the Step 5 commit-inventory (flags every non-pipeline commit) and the human `epic → main` PR + CI.

## 0.26.0 — ninth-audit remediation: immutable-OID gate+push, per-run frozen policy, assembly-bound paths, finalize-before-push
v0.25 bound the gate to the committed commit, but the colleague's ninth pass found two more mechanical false-green paths plus two flow defects. All reproduced before fixing.
- **P0#1 — gate and push are pinned to one immutable OID (no TOCTOU).** The gate checked a symbolic ref and the push re-resolved it, so the feature ref could move to an unverified commit B between check and push, sending B into the epic (reproduced with a local remote: a `ref:epic` push sent the moved tip). `run.md` Step 4 now resolves `VERIFIED_OID=$(git rev-parse "$TIP_REF")` **once**, gates that OID (`epic-gate.py --feature-ref "$VERIFIED_OID"`), and pushes that OID to both the feature branch and the epic (`git push origin "$VERIFIED_OID:…"`). New `tests/t_immutable_oid.sh` proves a pinned-OID push sends the verified commit while a ref push sends the moved tip.
- **P0#2 — the `[review]` policy is frozen for the whole run.** `merge-ledger.py` unconditionally overwrote `policy_hash` with the current policy, so a mid-run swap to a permissive policy downgraded a prior `high` block to advisory → green (reproduced). `merge-ledger.py` now records `policy_hash` on the first round and **refuses to overwrite** it on a later round whose policy differs — it exits non-zero so the orchestrator **parks** and requires a fresh full review; the frozen policy proceeds normally. New `[policy_freeze]` check executes it; `run.md` 2c parks on the non-zero exit.
- **Path correctness (parallel) — ledger/policy bound to the assembly worktree.** In parallel mode `merge-ledger.py "$LEDGER"` used a path relative to the shell cwd while the commit used `git -C "$ASSEMBLED"`, so the written and committed ledgers could diverge. `run.md` 2c now binds `LEDGER="$ASSEMBLED/$REL_LEDGER"` and `POLICY="$ASSEMBLED/.parallax/codex.toml"`, and commits the receipt with the relative path under `$ASSEMBLED` — everything rooted at the worktree actually under review.
- **Flow — finalize the receipt before pushing the feature.** The feature was pushed *before* the completion receipt was committed, then a CAS commit added the receipt locally, so the remote feature lagged the commit that entered the epic. Step 4 is reordered: finalize receipt → pin OID → push feature at the OID → gate → advance epic at the same OID, so the remote feature equals exactly what the gate checks and what enters the epic.
- Harness → **50 executable/presence checks** (47 → 50: +`[policy_freeze]`, +`t_immutable_oid.sh`, +the finalize/OID wiring greps). All prior checks pass; ZIP = `git archive` of the tag (no `.pyc`).
- **Honest scope & residual.** P0#2 (policy freeze) is mechanical (`merge-ledger.py`, harness-executed). P0#1 (immutable OID), the assembly-bound paths, and the finalize-before-push order are git **directives** in `run.md` executed by the orchestrator — the harness executes the exact git mechanic each relies on (`t_immutable_oid.sh`) and checks the contract wires it, but cannot prove the model runs the steps in order. The inherent residual is unchanged: a perfectly self-consistent forged feature commit is covered by the Step 5 commit-inventory and the human `epic → main` PR + CI.

## 0.25.0 — eighth-audit remediation: committed policy, frozen slice-set, CAS-finalized receipt, full slug identity
v0.24 bound the gate to the committed commit, but the colleague's eighth pass found two more false-green paths, a default-parallel-flow defect, and an identity gap. All reproduced before fixing.
- **P0#1 — the gate uses the COMMITTED policy, pinned by `policy_hash`.** The gate read run-state/ledgers from the feature ref but loaded `[review]` policy from the **working tree** (reproduced: one committed `high` blocker → strict policy holds, a permissive working-tree policy → verified). Now `scripts/epic-gate.py` reads `.parallax/codex.toml` via `git show <ref>:…` (never the worktree), and each ledger carries a `policy_hash` (new `scripts/merge-ledger.py --policy`, helper `triage.policy_hash`) that the gate requires to equal the committed policy's hash — so neither a working-tree swap (gate ignores it) nor a *committed* policy swapped in after review (hash mismatch ⇒ hold) can re-triage a receipt under a softer policy.
