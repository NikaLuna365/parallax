# Changelog

All notable changes to the Parallax plugin. Versions are cumulative.

## 0.26.0 — ninth-audit remediation: immutable-OID gate+push, per-run frozen policy, assembly-bound paths, finalize-before-push
v0.25 bound the gate to the committed commit, but the colleague's ninth pass found two more mechanical false-green paths plus two flow defects. All reproduced before fixing.
- **P0#1 — gate and push are pinned to one immutable OID (no TOCTOU).** The gate checked a symbolic ref and the push re-resolved it, so the feature ref could move to an unverified commit B between check and push, sending B into the epic (reproduced with a local remote: a `ref:epic` push sent the moved tip). `run.md` Step 4 now resolves `VERIFIED_OID=$(git rev-parse "$TIP_REF")` **once**, gates that OID (`epic-gate.py --feature-ref "$VERIFIED_OID"`), and pushes that OID to both the feature branch and the epic (`git push origin "$VERIFIED_OID:…"`). New `tests/t_immutable_oid.sh` proves a pinned-OID push sends the verified commit while a ref push sends the moved tip.
- **P0#2 — the `[review]` policy is frozen for the whole run.** `merge-ledger.py` unconditionally overwrote `policy_hash` with the current policy, so a mid-run swap to a permissive policy downgraded a prior `high` block to advisory → green (reproduced). `merge-ledger.py` now records `policy_hash` on the first round and **refuses to overwrite** it on a later round whose policy differs — it exits non-zero so the orchestrator **parks** and requires a fresh full review; the frozen policy proceeds normally. New `[policy_freeze]` check executes it; `run.md` 2c parks on the non-zero exit.
- **Path correctness (parallel) — ledger/policy bound to the assembly worktree.** In parallel mode `merge-ledger.py "$LEDGER"` used a path relative to the shell cwd while the commit used `git -C "$ASSEMBLED"`, so the written and committed ledgers could diverge. `run.md` 2c now binds `LEDGER="$ASSEMBLED/$REL_LEDGER"` and `POLICY="$ASSEMBLED/.parallax/codex.toml"`, and commits the receipt with the relative path under `$ASSEMBLED` — everything rooted at the worktree actually under review.
- **Flow — finalize the receipt before pushing the feature.** The feature was pushed *before* the completion receipt was committed, then a CAS commit added the receipt locally, so the remote feature lagged the commit that entered the epic. Step 4 is reordered: finalize receipt → pin OID → push feature at the OID → gate → advance epic at the same OID, so the remote feature equals exactly what the gate checks and what enters the epic.
- Harness → **50 executable/presence checks** (47 → 50: +`[policy_freeze]`, +`t_immutable_oid.sh`, +the finalize/OID wiring greps). All prior checks pass; ZIP = `git archive` of the tag (no `.pyc`).
- **Honest scope & residual.** P0#2 (policy freeze) is mechanical (`merge-ledger.py`, harness-executed). P0#1 (immutable OID), the assembly-bound paths, and the finalize-before-push order are git **directives** in `run.md` executed by the orchestrator — the harness executes the exact git mechanic each relies on (`t_immutable_oid.sh`) and checks the contract wires it, but cannot prove the model runs the steps in order. The inherent residual is unchanged: a perfectly self-consistent forged feature commit is covered by the Step 5 commit-inventory and the human `epic → main` PR + CI.

## 0.25.0 — eighth-audit remediation: committed policy, frozen slice-set, CAS-finalized receipt, full slug identity
v0.24 bound the gate to the committed commit, but the colleague's eighth pass found two more false-green paths, a default-parallel-flow defect, and an identity gap. All reproduced before fixing.
- **P0#1 — the gate uses the COMMITTED policy, pinned by `policy_hash`.** The gate read run-state/ledgers from the feature ref but loaded `[review]` policy from the **working tree** (reproduced: one committed `high` blocker → strict policy holds, a permissive working-tree policy → verified). Now `scripts/epic-gate.py` reads `.parallax/codex.toml` via `git show <ref>:…` (never the worktree), and each ledger carries a `policy_hash` (new `scripts/merge-ledger.py --policy`, helper `triage.policy_hash`) that the gate requires to equal the committed policy's hash — so neither a working-tree swap (gate ignores it) nor a *committed* policy swapped in after review (hash mismatch ⇒ hold) can re-triage a receipt under a softer policy.
- **P0#2 — run-state's slice set must EQUAL the frozen manifest.** The gate trusted whatever slices run-state listed, so a run-state that omitted a slice still verified (reproduced: committed `slices.md` had S1+S2, run-state only S1 → verified). New machine-readable `.parallax/<slug>/slices.lock` (frozen at spec-freeze; `assets/slices-lock.schema.json`); the gate requires the run-state slice set to equal `slices.lock` exactly — a dropped or added slice ⇒ hold.
- **P1#3 — the completion receipt is finalized onto the feature ref (parallel-safe).** Autonomous/parallel leaves `$ROOT` **detached**, so the v0.24 finalization's plain `git commit` landed on a dangling detached HEAD, never on `feature/<slug>` — the gate then read `status≠complete` and a *correct* parallel run always got **hold** (reproduced: `receipt_is_on_feature=no`). `run.md` Step 4 now builds the receipt in a transient detached worktree and advances the branch by a **CAS `update-ref`**, exactly like parallel integration. New `tests/t_finalize.sh` executes it with `$ROOT` detached.
- **P1#4 — full slug identity.** The gate checked `slice_id` but not `slug`, so another feature's receipts (same slice ids) under a different path were accepted (reproduced). The gate now requires `run-state.slug`, each `ledger.slug`, and `slices.lock.slug` all to equal the `--slug` being promoted.
- New `scripts/code-tree-hash.sh` consumer + `scripts/slices-lock` schema + `tests/t_epic_gate.sh` (rewritten, 11 scenarios) + `tests/t_finalize.sh`. Harness → **47 executable/presence checks** (46 → 47). All prior checks pass; ZIP = `git archive` of the tag (no `.pyc`).
- **Honest scope & residual.** The gate is mechanical and now bound to the committed policy, the frozen slice-set, the per-feature slug, and the promoted tree. The remaining residual is unchanged and inherent: a *perfectly self-consistent forged* feature commit (forged code + run-state + slices.lock + ledgers + a matching committed policy, all mutually agreeing) cannot be distinguished from a genuine one without re-running the verifier — covered by the Step 5 commit-inventory (flags every non-pipeline commit) and by `epic → main` always being a human PR + CI. Producing `slices.lock`, recording `policy_hash`, and CAS-finalizing the receipt are orchestrator/spec **directives** in `run.md`/`spec.md`; the gate that consumes them is mechanical and harness-executed.

## 0.24.0 — seventh-audit remediation: a feature-level receipt bound to the actual promoted commit
v0.23's `epic-gate.py` validated free-floating JSON in the working tree, not the commit being promoted. The colleague's seventh pass showed the gate could pass on an uncommitted ledger, on code changed after review, on an operator-narrowed slice list, and on a receipt whose identity/round-count was bogus — plus a `.pyc` shipped in the release. All reproduced before fixing; the gate is now rewritten to read **only** from the committed feature ref and to bind its verdict to that exact commit.
- **P0#1 — the gate reads committed receipts and binds to the feature tree.** It read ledgers from the working tree and trusted the diff inside the ledger; no git ref was passed (reproduced: code changed after review → still verified; an uncommitted ledger → verified). `scripts/epic-gate.py` is rewritten to take `--feature-ref` and read `run-state.json` + every ledger via `git show <ref>:…` (never the worktree), and to require `run-state.verified_tree` to equal the **recomputed** code-tree hash of the promoted commit (new `scripts/code-tree-hash.sh` — a hash of every tracked non-`.parallax/` file, the same canonical hash `run.md` records at completion). A code/test/config change after the run completed moves that hash ⇒ **hold**. New `run-state.schema.json` field `verified_tree` (required once `status=complete`).
- **P0#2 — the slice set comes from the validated run-state, not a CLI arg.** `--slices` was a free argument and `run.md` left a placeholder, so an operator could list only the verified slices and omit a parked/unverified one (reproduced). The gate now reads the slice list from the **committed** `run-state.json`, requires `status = complete`, and requires **every** slice to be `integrated` (no `parked`/`pending`/`green-unverified`). No `--slices`, no `--reviews-dir`.
- **P1#3 — receipt identity + a real verifier round are enforced.** A ledger file `S2.json` carrying `"slice_id":"S1"` was accepted, and `rounds_used:0` (zero verifier invocations) passed (both reproduced). The gate now requires each ledger's `slice_id` to match the slice it is read for, and `rounds_used ≥ 1`.
- **P2 — no compiled bytecode ships.** `scripts/__pycache__/triage.cpython-311.pyc` had been committed (swept in by a `git add -A` during packaging). Removed from git; `.gitignore` now excludes `__pycache__/` + `*.pyc`/`*.pyo`; the harness runs with `PYTHONDONTWRITEBYTECODE=1`; new `[no_pyc]` check asserts none is tracked/shipped.
- New scripts `scripts/epic-gate.py` (rewritten) + `scripts/code-tree-hash.sh`; new tests `tests/t_epic_gate.sh`. Harness → **46 executable/presence checks** (45 → 46). All prior checks still pass; ZIP is `git archive` of the tag (tracked files only — no `.pyc`).
- **Honest scope & residual.** The gate is **mechanical** (executed by `epic-gate.py` against real git repos, locked by `t_epic_gate.sh`): it proves the receipts are committed, internally consistent, identity-correct, and that the promoted commit's code matches the recorded `verified_tree`. What it does *not* do — and cannot, without re-running the verifier — is prove a *perfectly self-consistent forged* feature commit (forged code + run-state + ledgers all mutually agreeing) wasn't hand-built; that residual is covered by the Step 5 commit-inventory flagging every non-pipeline commit for human review, and by `epic → main` always being a human PR + CI. Recording `verified_tree` and committing run-state at completion is an orchestrator **directive** (`run.md`); the gate that consumes it is mechanical.

## 0.23.0 — sixth-audit remediation: the promoted commit, the integrated ledger, and a computed epic gate
v0.22 fixed the prior five, but the colleague's sixth pass found that an autonomous feature → epic — *especially in parallel mode* — could still admit a false green four ways and close the wrong finding a fifth. All five reproduced in a sandbox before fixing, then closed (executable regressions, or a presence check honestly labeled where the action is a contract directive).
- **P0#1 — the promoted commit is now EXACTLY the reviewed tree + receipt (no `git add -A`).** `git diff --quiet` ignores untracked files, but the green path then ran `git add -A`, sweeping any untracked, **un-reviewed** file into the promoted commit (reproduced: committed tree ≠ reviewed tree). Fixed in `run.md` 2c: the reviewed scope is guarded for **both** unstaged tracked changes **and** untracked files (`git ls-files --others`), the review diff is a content hash of **exactly** the reviewed code+tests (`git ls-files -s | git hash-object`, never `HEAD^{tree}`), and the green commit stages **only** the receipt (`git add -- "$LEDGER"`) and commits the already-staged index — never `git add -A`. (Also fixes a latent `git add -- <glob>` fatal on an empty test glob.) New `t_difftree.sh` executes it.
- **P0#2 — the review ledger now rides into the parallel CAS commit.** Parallel integration re-applied only `src`/`tests` deltas onto the new tip, so the committed `.parallax/<slug>/reviews/<slice>.json` (memory, round budget, codex proof) was **dropped** (reproduced: green commit has ledger, integrated tip has none). Fixed in `run.md`: the integration applies one delta **from the green commit** over `src`+`tests`+`.parallax/<slug>/reviews/`, so the receipt is carried and *what integrates is exactly what was verified*. New `t_parallel_ledger.sh` executes it.
- **P1#3 — a cited `id` can no longer close a different finding.** `resolved[].id` was honored without checking it matched (reproduced: a `resolved` quoting a safety finding's id but carrying unrelated metadata marked it fixed → triage green). `merge-ledger.py` now honors a cited id (on `findings` and `resolved`) **only if** it agrees with that finding's fingerprint (`kind|spec_ref|file`); a mismatch falls back to fingerprint matching (fail-safe — it can't settle the cited finding). New `[id_consistency]` check.
- **P1#4 — the epic gate is computed mechanically, not a free variable.** `PARALLAX_VERIFIED` was only *read*, never computed, so a preset/stale `1` lifted the hold. New `scripts/epic-gate.py` computes verification from the **committed** per-slice receipts: every integrated slice must have a committed ledger that triages GREEN (each blocker codex-verified against its reviewed tree) within budget — a missing ledger (e.g. `on_missing=warn` produced none) or any live blocker ⇒ **hold**. It reads policy only from the trusted toml and validates each ledger fail-closed. `run.md` Step 4 gates the epic push on `epic-gate.py` (no env override). New `[epic_gate]` check executes it (incl. that a preset env var can't lift the hold).
- **P1#5 — the unstaged-changes guard no longer wedges the next review.** The guard was repo-wide, so a modified **tracked** ledger (review metadata that changes every round) made `git diff --quiet` non-zero and the next round aborted as "unstaged changes" even with correct code (reproduced). The guard is now scoped to the reviewed `src`/`tests` pathspecs, excluding `.parallax/`. New `t_difftree.sh` proves a ledger-only change doesn't trip it.
- **Convergence guard (found while fixing the above).** Because the review diff is now content-sensitive, a fix confirmed in an early round must keep tracking the latest tree across rounds; `merge-ledger.py` now **re-stamps** an already-`fixed` finding's `last_verified_diff` when the verifier re-confirms it (via a consistent cited id) against the current tree — so a multi-round slice converges to green instead of falsely blocking. New `[ledger_restamp]` check.
- New script `scripts/epic-gate.py`; harness → **45 executable/presence checks** (40 → 45). All prior checks still pass.
- **Honest scope:** P1#3, P1#4, and the convergence guard are **mechanical** (executed by `merge-ledger.py` / `epic-gate.py`, locked by the harness). P0#1, P0#2, P1#5 are git **directives** in `run.md` executed by the orchestrator — the harness executes the exact git mechanic each one relies on (`t_difftree.sh`, `t_parallel_ledger.sh`) and checks the contract wires it, but cannot prove the model runs it. Same honesty split as `mode`/`on_missing`.

## 0.22.0 — fifth-audit remediation: make feature → epic autonomous-safe (close the remaining false-green paths)
v0.21 closed the producer-self-certification holes, but the colleague's fifth pass found that an autonomous feature → epic promotion could still admit a false green four ways. All five findings were **reproduced in a sandbox before any fix** (see the repro that drove each regression test), then closed with an executable (or, for the directive-level item, presence) regression.
- **P0#1 — certify the CURRENT staged tree, not a stale `HEAD^{tree}`.** The post-green review diff was `git rev-parse HEAD^{tree}`, but the assembled slice is **staged, not committed** — HEAD hasn't moved, so a codex-verified `fixed` recorded against `HEAD^{tree}` re-matched a tree that had since changed ⇒ **false green** (reproduced: content changed A→B, `triage` still returned `green`/exit 0). Fixed in `run.md`: the diff is now `git -C "$ASSEMBLED" write-tree` (a content hash of the actual staged index), guarded by `git -C "$ASSEMBLED" diff --quiet` so the SHA is exactly what the verifier read on disk. New `tests/t_difftree.sh` EXECUTES the git mechanic: HEAD^{tree} is identical across the A→B change (false-greens) while write-tree differs (re-blocks the stale fix).
- **P0#2 — a Codex `pass` no longer bypasses the ledger.** The GREEN path committed immediately on `pass`; `merge-ledger`+`triage` ran only on `concerns`. So a bare `pass` that simply omitted a still-open prior `high`/`safety` finding shipped green (reproduced: routing that pass through merge+triage yields `escalate`, not green). Fixed in `run.md`: **both** `pass` and `concerns` now fold the round into the per-slice ledger and let `triage.py` judge the **whole** ledger — any prior `open`/`regressed` finding the verifier didn't positively `resolved` is still live and still blocks. (The old standalone commit-on-pass code block is gone; bash-block count 14→13.) New `[pass_through_ledger]` harness check executes it.
- **P0#3 — the triage validator now fails CLOSED.** With `jsonschema` absent, `triage.py` skipped validation and returned `green` on a `{}` ledger (reproduced in a clean env: `green`/exit 0). For an autonomous gate, *inability to validate* must not pass as green: a missing validator, a missing schema file, or a schema violation now each ⇒ `escalate` (exit 2). The **only** skip is the explicit, insecure `--no-schema-check` (logic-isolation tests). `scripts/cloud-setup.sh` now treats `jsonschema` as **required** and aborts setup if it can't be imported (so the operator learns now, not mid-run). New `[review_failclosed]` check runs triage with the validator hidden and asserts `escalate`.
- **P1#4 — fingerprints no longer collapse distinct defects.** The merge keyed solely on `sha256(kind|spec_ref|file)`, so two *different* violations of the same section in the same file merged into one ledger entry and the first was **lost** (reproduced: 2 findings in → 1 stored). `merge-ledger.py` now matches two-pass: a round finding/resolution may cite an **exact ledger `id`** (binding precisely even when fingerprints collide), and uncited findings take the first *still-unconsumed* same-fingerprint slot else become a new distinct id — so two same-section defects never collapse. `review-round.schema.json` gains an optional `id` on `findings` and `resolved`; `role-codex-judge` is told to echo it. The cross-round rephrase→same-id regression behavior is unchanged (still tested). New `[merge_ledger_collision]` check.
- **P1#5 — an UNVERIFIED run cannot advance the append-only epic.** Under `on_missing = "warn"` the run proceeded stamping outputs `UNVERIFIED`, but Step 4 still auto-advanced feature → epic. Fixed in `run.md`: a hard verification gate (`PARALLAX_VERIFIED`) precedes the epic push — an UNVERIFIED run may push the **feature** branch for human review but **parks an epic-hold** and never advances the append-only epic; only a fully verified feature advances automatically. (`epic → main` remains a human PR + CI, as before.) New `[epic_verify_gate]` presence check; this guard is a **directive** executed by the orchestrator (honestly labeled), with a concrete bash gate that `bash -n` validates.
- Harness → **40 executable/presence checks** (32 → 40; +8 for the five findings above). All prior checks still pass.
- **Honest scope:** P0#1/#2/#3 and P1#4 are **mechanical** (executed by scripts/git and locked by the harness). P1#5 is a **directive** the orchestrating model must honor — the harness checks the gate is present and bash-valid, not that the model runs it; the same honesty split as `on_missing` and `mode` semantics.

## 0.21.0 — fourth-audit remediation: close the producer-self-certification holes
The v0.20 review ledger was written by the orchestrating Claude and triaged with a ledger-supplied policy — so a crafted ledger could mechanically turn a `high safety` finding into `green`. v0.21 makes certification something only the *independent* model can grant. (All findings reproduced before fixing.)
- **P0 — policy comes only from trusted config.** `triage.py` now reads the `[review]` policy **only** from the `.parallax/codex.toml` passed via `--policy`, **never** from the ledger (a ledger `policy` with `always_block_kinds=[]` previously waved a `safety` finding through). Missing/unparseable config ⇒ strict built-in defaults (fail-closed). The ledger schema also **rejects** any `policy` key (second layer). *(reproduced: high-safety → green; now → block.)*
- **P0 — a fix must be proven, not asserted.** A `fixed` finding counts as settled **only** if `verified_by == "codex"` **and** `last_verified_diff == --current-diff` (the actual assembled tree). A `fixed` Claude merely stamped, or one verified against a stale tree, is treated as **live** and still blocks. Enforced in `triage.py` **and** in the ledger schema (status=fixed ⇒ requires `verified_by:"codex"` + a hex `last_verified_diff`). *(reproduced: faked `fixed` → green; now → block.)*
- **P0 — unified review-round + mechanical merge (the producer authors nothing).** New `assets/codex/review-round.schema.json` is what the verifier emits (`findings` + positively-re-verified `resolved`). New `scripts/merge-ledger.py` is the **only** writer of the ledger: it maps the round in by **fingerprint** (`sha256(kind|spec_ref|file)` → the same defect keeps one id across rounds), assigns ids, and sets `verified_by=codex` only on the verifier's `resolved` items. Claude never invents `spec_ref`/`evidence`/`id`/lifecycle. Resolves the old verdict↔ledger shape mismatch and the missing finding lifecycle.
- **P1 — per-slice ledgers, single-sourced budget.** Ledgers moved to `.parallax/<slug>/reviews/<slice_id>.json` (one per slice — a slice can't spend another's budget). `rounds_used` lives **only** in the ledger (run-state points at the file via `reviews_dir`; the divergent `review_rounds` counter is gone). Round semantics pinned: one `merge-ledger` call = one round, the initial review is round 1, so `max_rounds=2` ⇒ at most two verifier invocations.
- **P1 — parallel correctness.** The arbiter **and** the post-green verifier now run with cwd = the slice's `$WT/S<n>/assembly` (the tree actually under review), never `$ROOT`. And `feature/<slug>` is **never checked out** in a persistent worktree during a wave (`$ROOT` detached) — moving the branch ref via CAS while it was checked out left that worktree stale/dirty (`D src/…`); reproduced and fixed.
- **P1 — expired-lock steal is now race-safe.** Stealing an expired lease uses `--force-with-lease=<ref>:<observed-oid>` (a bare `--force` let *two* stealers both win — reproduced), and release is **fenced** to your own lock oid so you never clobber a successor. `t_lock.sh` executes both races.
- **P2 — verdict/round consistency.** `verdict.schema.json` and `review-round.schema.json` now reject `pass`+non-empty-findings and `concerns`+empty-findings.
- Harness → **32 executable checks** (added: the four reproduced exploits as regressions — ledger-policy ignored, faked-fixed blocks, codex-verified-vs-current-diff settles, stale-verify re-blocks; `merge-ledger` fingerprint/lifecycle; schema rejects fixed-without-proof + policy-bearing ledger + verdict inconsistency; expired-lock lease steal + fenced release).

## 0.20.0 — verifier memory: anti-oscillation review ledger + bounded hybrid disposition
Addresses the cross-model verifier's missing memory between review rounds (it re-discovered fixed findings, never confirmed a fix held, and any `low` nit could stall the whole dependent DAG).
- **Review memory is a ledger, not a session.** Each review is a **fresh** verifier (`[review] resume_codex_session = false`); memory lives in a branch-committed `.parallax/<slug>/review-ledger.json` (new `assets/codex/review-ledger.schema.json`). A fresh verifier re-checks prior `open`+`fixed` findings for **regression first**, then scans for new ones — killing the `n → fix → m → variant-of-n` loop **without** the anchoring a persistent `codex exec resume` session causes (and the non-interactive session id is fragile to capture and wouldn't survive a cloud fresh-clone). Every finding must carry a **`spec_ref` + concrete evidence**; style/preference is not a defect. (Verified: `codex exec resume`/`--last` exist — we deliberately don't use them.)
- **Disposition is mechanical, not editorial — `scripts/triage.py`.** A new, unit-tested pure function decides `green | block | escalate` from the ledger + `[review]` policy: `low` = **advisory** (recorded, non-blocking); `medium`/`high` **block**; `safety`/`anti-cheat`/`spec-gap` and any **reproducible functional error** **always** block. Claude may **contest** a blocker only via a formal, ledgered reason (`duplicate`/`not-reproducible`/`contradicts-spec`/`out-of-scope`) — a contested medium/high **escalates** (human decides), it is **never silently dropped or auto-greened**. So the asymmetry that mattered (Claude can't quietly overrule the verifier on anything that blocks) is preserved; the only relaxation is the `low`-advisory release valve.
- **Bounded loop.** `[review].max_rounds` (default 2) caps review rounds per slice (tracked as `review_rounds` in the checkpoint, distinct from the 3-iteration worker breaker); at the cap with blockers still live, the slice **parks** instead of looping on ever-smaller nits.
- Contracts updated: `role-codex-judge` (the ledger protocol + hybrid disposition, honestly split into mechanical vs directive), `run.md` Step 2c (*Review memory, rounds & disposition* — ledger merge → triage → block/escalate/green), `parallax-core` (the no-quiet-overrule nuance), run-state (`review_rounds` per slice + `review_ledger` queue path), README.
- Harness → **30 executable checks** (added: `triage.py` disposition matrix incl. exit codes, review-ledger schema rejects missing `spec_ref`/bad kind/bad rebuttal, `[review]` config semantics, contract-presence).

## 0.19.0 — third-audit remediation (cloud-lock race, transactional parallel integration, exact-resume schema)
- **#1 (P0 cloud lock, fixed):** two fresh same-`HEAD` clones used to *both* win the lock — pointing the ref at `HEAD` made the loser's "create" a no-op same-value push. The lock now points at a **unique lock commit** (`git commit-tree` with the `run_id` baked in) and is taken create-if-absent locally / by **`git push --force-with-lease=<ref>:`** in the cloud (push only if the ref is absent on origin). `t_lock.sh` now models the *real* same-HEAD race and includes a guard proving the old same-value approach let both win.
- **#2/#3 (P1 parallel integration, fixed):** each slice integrates in its **own `$WT/S<n>/assembly` worktree** (not the shared `feature/<slug>` tree), **transactionally** — both track patches apply all-or-nothing; a conflict triggers `git reset --hard` so the assembly is abandoned and `feature/<slug>` is **never** left half-patched. Advancing `feature/<slug>` is a **CAS `update-ref`** (old-value = the tip we forked from), serializing concurrent integrations; a slice that loses the race re-applies its `wave_base` diff onto the new tip. New `t_conflict.sh` executes the conflict → rollback → feature-unmoved path.
- **#4 (P1 binary files, fixed):** integration uses **`git diff --binary … | git apply --3way --index --binary`** — a plain text diff of a binary file fails (`cannot apply binary patch without full index line`). New `t_binary.sh` proves the `--binary` path applies (bytes match) and the plain path fails.
- **#5 (P1 exact resume, fixed):** run-state schema now records **`wave_base`** per slice (required once `in_progress`/`green-unverified`), constrains all object ids (`base_tip`/`code_tip`/`test_tip`/`wave_base`/`verified_diff`) to a **hex-SHA pattern**, and conditionally requires **`lock` when `status=running`** and **`paused` when `status=paused-on-limit`**. The harness validates a complete checkpoint and rejects 5 distinct incomplete/invalid ones.
- **#6 (P1 sole verdict, fixed):** the verdict schema gains **`code-fault`/`test-fault`** finding kinds, and `role-codex-judge` documents **sole-mode RED arbitration** (the verifier classifies the fault, a third behavior beyond pre-freeze/post-green) so `mode = sole` is actually expressible in the schema it emits.
